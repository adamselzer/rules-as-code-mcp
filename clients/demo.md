# Connecting an MCP client to `rules-as-code-mcp`

The server speaks the Model Context Protocol over **stdio** (how Claude Desktop and
Claude Code connect) and **streamable-http** (the production path). Below: how to
run the captured demo, how to wire it into Claude Desktop / Claude Code, and how
scope works on each transport.

A full, real transcript produced by driving the server over stdio is in
[`demo_transcript.md`](demo_transcript.md). It was generated by
[`demo_client.py`](demo_client.py), not hand-written.

## Run the captured demo yourself

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"
python clients/demo_client.py      # launches the server twice (both roles), writes demo_transcript.md
```

The client launches the server as a subprocess in two roles and exercises every
tool, including the failure cases.

## Roles and scope

Two scopes, with caseworker a superset of screening:

| Scope | Tools | PII |
|---|---|---|
| `screening` | `screen_programs`, `lookup_policy` | none collected or stored |
| `caseworker` | all tools (determinations, verifications, explanations) | full case detail |

Scope is resolved from the **request context**, never from a tool argument the
model controls, so a model cannot escalate its own privileges:

- **stdio**: there is no bearer token, so the role comes from the `RULES_MCP_ROLE`
  environment variable (`screening` | `caseworker`), set when the client launches
  the server. Default is `screening`.
- **streamable-http**: scope comes from the OAuth bearer token, validated by
  `SimpleTokenVerifier` in [`server/auth.py`](../server/auth.py), which maps a token
  to its scopes.

## Wire it into Claude Desktop

Add to `claude_desktop_config.json` (Settings â†’ Developer â†’ Edit Config). Use an
absolute path to the project's Python so the dependencies resolve:

```json
{
  "mcpServers": {
    "rules-as-code": {
      "command": "/ABSOLUTE/PATH/TO/rules-as-code-mcp/.venv/bin/python",
      "args": ["-m", "server.main"],
      "cwd": "/ABSOLUTE/PATH/TO/rules-as-code-mcp",
      "env": { "RULES_MCP_ROLE": "caseworker" }
    }
  }
}
```

Restart Claude Desktop. The five tools appear under the server, and you can ask,
for example, *"Screen a household of three with $2,100/month earned income and
$1,200 in housing costs,"* then *"Now run a full SNAP determination and explain
which rules fired."*

## Wire it into Claude Code

```bash
claude mcp add rules-as-code \
  --env RULES_MCP_ROLE=caseworker \
  -- /ABSOLUTE/PATH/TO/.venv/bin/python -m server.main
```

(run from the project directory, or pass an absolute `cwd`).

## Run over HTTP instead

```bash
python -m server.main --http     # streamable-http on 127.0.0.1:8000
```

## Two layers of input validation (by design)

Bad input is rejected at two layers, and the transcript shows both:

1. **Schema layer (MCP/pydantic).** Tools declare typed inputs (e.g. `household:
   Household`), so the SDK publishes a precise JSON schema and rejects type- or
   shape-violations *before the tool body runs* (see the empty-`members` case in
   the transcript).
2. **Domain layer (structured errors).** Authorization, tool misuse, unknown ids,
   and semantic constraints raise the typed, recoverable errors in
   [`server/errors.py`](../server/errors.py), JSON-encoded so a model can parse and
   recover from them.

Neither layer ever returns a stack trace.
